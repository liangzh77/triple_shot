## 2. 产品目标

### 2.1 核心目标

构建一个轻量、可靠、响应快的应用，先把用户最常用的核心流程打磨清楚，再逐步扩展功能。

### 2.2 体验目标

- 打开即用：首屏直接进入主要工作区，而不是营销页。
- 信息清楚：用户能快速理解当前状态、可执行动作和下一步。
- 操作顺手：常用动作路径短，反馈及时，错误可恢复。
- 视觉可信：使用真实照片建立场景感，避免泛化、模板化的 UI。
- 本地可靠：技术设计上偏向简单、稳定、可维护的数据存储。

## 3. 技术方向

### 3.1 初步技术栈

- 语言：Go
- 数据库：SQLite
- 写入模式：WAL

### 3.2 初步技术取向

- 优先做单机/本地优先能力，减少部署和依赖复杂度。
- SQLite 开启 WAL，提高读写并发体验和数据安全性。
- 后端接口保持简单清晰，为后续桌面端、Web 端或移动端扩展留空间。
- 在 UI 设计定稿前，不急于确定数据库表结构和 API 细节。

## 5. 核心使用场景

待补充原始需求后细化。当前先按 UI 设计需要，预留以下场景类型：

1. 首次进入应用  
   用户需要快速理解应用用途，并看到可执行的第一步。

2. 浏览核心内容  
   用户需要查看列表、状态、分类、照片或关键记录。

3. 创建或编辑内容  
   用户需要低摩擦地新增、修改、保存，且能看到保存反馈。

4. 查看详情  
   用户需要进入单条记录/对象的详情页，看到关键信息、照片和可操作项。

5. 筛选与搜索  
   用户需要快速缩小范围，找到目标内容。

6. 空状态与错误恢复  
   用户没有数据、图片缺失、保存失败时，界面需要给出清楚反馈。

## 6. 信息架构草案

### 6.1 主要区域

- 主页/工作台：展示核心状态、近期内容和主要操作入口。
- 列表页：承载主要对象的浏览、筛选、排序和批量操作。
- 详情页：展示单个对象的完整信息、照片、历史和操作。
- 创建/编辑页：完成新增、编辑、保存、取消和校验。
- 设置页：承载数据路径、备份、主题、偏好等低频配置。

### 6.2 导航建议

- 桌面端：左侧窄导航 + 主内容区。
- 移动端：底部导航或顶部返回结构，保证核心动作易触达。
- 详情页使用清晰返回路径，避免用户迷失。

## 7. UI 设计方向

### 7.1 整体气质

关键词：

- 真实
- 清爽
- 有秩序
- 轻量但不简陋
- 有照片带来的温度

避免：

- 过度渐变
- 过度圆角
- 大量漂浮卡片
- 泛紫色科技风
- 只有装饰性、无法传达真实内容的图片

### 7.2 视觉基调

待照片确定后再定最终色板。初步原则：

- 从照片中提取 1 个主色、1 个强调色、2-3 个中性色。
- 背景保持干净，避免和照片抢注意力。
- 主要按钮使用高对比色，危险操作使用明确警示色。
- 文字层级克制，列表和表单优先保证可读性。

### 7.3 照片使用原则

照片不是装饰，而是帮助用户理解对象、场景和状态。

建议准备：

- 首屏主视觉照片：能代表产品/场景/对象本身。
- 列表缩略图：横竖比例统一，内容主体清楚。
- 详情页大图：适合展示纹理、环境或关键细节。
- 空状态照片或插图：真实但不过度抢戏。
- 背景氛围图：只在需要建立品牌感时使用，避免模糊暗图。

建议图片规格：

- 首屏图：至少 1600px 宽。
- 卡片/列表图：建议 4:3 或 1:1。
- 详情图：建议 16:9 或 3:2。
- 文件命名：`scene-home-01.jpg`、`item-thumb-01.jpg`、`detail-hero-01.jpg`。

## 8. 关键页面草案

### 8.1 主页/工作台

目的：让用户马上知道当前有什么、下一步能做什么。

建议内容：

- 顶部：应用名称、搜索入口、主要新增按钮。
- 主区域：最近内容、关键状态、推荐下一步。
- 视觉区：使用一张真实照片作为场景提示，但不遮挡核心操作。
- 低频区：最近修改、待处理项、数据同步/备份状态。

核心互动：

- 点击主要按钮进入创建流程。
- 点击最近内容进入详情。
- 搜索框支持即时建议。
- 空数据时显示明确的创建入口。

### 8.2 列表页

目的：高效浏览、比较、筛选内容。

建议内容：

- 搜索
- 筛选
- 排序
- 列表/网格视图切换
- 每项包含缩略图、标题、状态、更新时间和快速操作

核心互动：

- 支持键盘方向键/回车进入详情。
- 筛选条件可清除。
- 图片加载失败时显示稳定占位。
- 长列表需要保持滚动和选择状态。

### 8.3 详情页

目的：展示单个对象的完整上下文，并支持下一步操作。

建议内容：

- 顶部大图或照片组。
- 标题、状态、关键字段。
- 操作栏：编辑、删除、归档/完成等。
- 关联记录或历史变化。

核心互动：

- 图片可查看大图。
- 编辑后返回详情并显示保存成功反馈。
- 删除类操作需要确认。
- 返回列表时保留之前筛选和滚动位置。

### 8.4 创建/编辑页

目的：让用户快速录入高质量信息。

建议内容：

- 必填字段清楚标识。
- 图片上传/选择区域。
- 保存、取消、预览。
- 表单校验错误靠近字段显示。

核心互动：

- 自动保存草稿可作为后续增强。
- 保存中显示按钮加载状态，防止重复提交。
- 取消时如有未保存变更，需要二次确认。

### 8.5 设置页

目的：放置低频但重要的控制项。

建议内容：

- 数据库位置/数据文件状态。
- 备份与恢复。
- 外观主题。
- 导入导出。
- 关于应用。

## 10. 互动原则

- 所有用户动作都要有即时反馈。
- 保存、删除、导入、恢复等操作要有明确成功/失败状态。
- 主要操作路径不超过 2-3 步。
- 危险操作必须可理解、可取消。
- 键盘和鼠标都要能完成核心流程。
- 移动端触控目标不小于 44px。
- 加载状态优先用骨架屏，少用单独转圈。
